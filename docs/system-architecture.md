# 轻量级安全建模与静态攻击路径分析系统架构设计

## 1. 目标与边界

本系统定位为**工程分析与审查工具**，而不是传统 CRUD 后台。

- 建模对象：AssetNode、AssetEdge、ThreatPoint、AttackPath。
- 分析性质：静态图结构可达性分析与启发式优先级评估。
- 数据权威：Neo4j 为唯一真源（Single Source of Truth）。
- 前端职责：图形化建模、审查、交互式推演展示；不持久化业务真值。

不引入完整 ATT&CK/STRIDE 流程语义，不做动态仿真或概率预测。

## 2. 整体架构与职责划分

```text
┌─────────────────────────────────────────────────────────────┐
│                         前端 (Web)                         │
│  图建模画布 / 关系编辑 / ThreatPoint 挂载 / 路径审查       │
│  Excel 导入向导 / 差异预览 / 显式保存 / 解释性评分展示      │
└───────────────▲───────────────────────────────┬────────────┘
                │ REST/Graph API               │
                │ (草稿态 + 提交态)             │
┌───────────────┴───────────────────────────────▼────────────┐
│                        后端服务层                           │
│  1) Schema/语义校验                                          │
│  2) Excel 解析与字段映射引擎                                 │
│  3) 图变更集(Delta)管理与事务提交                            │
│  4) 攻击路径推演(多跳遍历)                                   │
│  5) 启发式评分与解释生成                                     │
│  6) 审计日志与版本快照                                       │
└───────────────▲───────────────────────────────┬────────────┘
                │ Cypher / Bolt                │
┌───────────────┴───────────────────────────────▼────────────┐
│                       Neo4j (权威库)                         │
│  AssetNode / AssetEdge / ThreatPoint / AttackPath            │
│  索引约束 / 图查询 / 路径持久化 / 历史版本标签               │
└─────────────────────────────────────────────────────────────┘
```

### 2.1 前端职责

1. **图建模 UI 与交互编辑**：节点/边新增、删除、属性修改。
2. **ThreatPoint 挂载管理**：在资产详情中静态挂载/解除挂载。
3. **攻击路径展示**：多跳路径可视化、按评分排序、解释可读化。
4. **Excel 导入流程**：上传 → 预校验 → 字段映射确认 → 差异预览。
5. **显式保存机制**：所有修改先在前端“工作区草稿”中，用户点击保存后统一提交。

### 2.2 后端职责

1. **语义与约束守卫**：校验核心实体关系是否符合轻量语义模型。
2. **导入映射引擎**：将 Excel 列映射到标准实体字段并做类型/枚举校验。
3. **变更集事务化提交**：将前端草稿转换为原子变更集，在 Neo4j 中一次性提交。
4. **路径推演服务**：以 ThreatPoint 所在 AssetNode 为起点进行多跳遍历。
5. **评分解释服务**：输出启发式分值与“为何高优先级”的解释条目。
6. **快照与审计**：记录提交人、时间、变更范围、路径结果版本。

### 2.3 Neo4j 职责

1. **单一真源存储**：持久化全部核心实体与关系。
2. **图查询执行**：提供路径查询、邻接检索、子图过滤。
3. **约束与索引**：确保业务主键唯一性与查询性能。
4. **分析结果落库**：AttackPath 作为分析工件可持久化并关联生成批次。

## 3. 语义模型建议（最小闭环）

> 仅示意，不扩展到复杂框架术语。

### 3.1 节点类型

- `AssetNode`
  - `assetId`（唯一）
  - `name`
  - `type`
  - `criticality`（用于启发式评分）
  - `owner`
  - `tags[]`

- `ThreatPoint`
  - `threatId`（唯一）
  - `name`
  - `category`（轻量枚举）
  - `preconditionText`（可选）
  - `severityBase`（启发式基准）

- `AttackPath`
  - `pathId`（唯一）
  - `hopCount`
  - `score`
  - `priority`（P1/P2/P3）
  - `explanations[]`
  - `generatedAt`
  - `analysisVersion`

### 3.2 关系类型

- `(:AssetNode)-[:CONNECTED_TO {edgeId, relationType, trustBoundary, directionality}]->(:AssetNode)`
- `(:ThreatPoint)-[:MOUNTED_ON]->(:AssetNode)`
- `(:AttackPath)-[:STARTS_FROM]->(:ThreatPoint)`
- `(:AttackPath)-[:REACHES]->(:AssetNode)`（可按 hop 序号附属性）
- `(:AttackPath)-[:USES_EDGE]->(:AssetEdgeRef)`（可选：若将边路径拆成中间对象）

> `AssetEdge` 若需作为一等实体，可采用“关系实体化”模式（节点化 + 两段关系），便于挂载审计与评分特征。

## 4. 核心数据流设计

## 4.1 Excel 批量导入流

1. 用户上传 Excel（可多 Sheet：Assets / Edges / ThreatPoints）。
2. 后端读取模板版本，执行字段映射校验。
3. 返回预检查报告：
   - 必填缺失
   - 主键冲突
   - 引用不存在（如 Edge 端点资产不存在）
4. 前端展示“映射确认 + 差异预览（新增/更新/删除）”。
5. 用户确认后触发 `commitImport`。
6. 后端开启事务写入 Neo4j，成功后返回变更摘要与版本号。

## 4.2 图编辑与显式保存流

1. 前端画布操作仅更新本地工作区（Draft Graph）。
2. 前端维护 `changeSet`（add/update/delete）。
3. 用户点击“保存”后一次性提交 `changeSet`。
4. 后端进行语义校验 + 冲突检测（可选乐观锁 version）。
5. 校验通过则事务写入 Neo4j，并返回新版本与审计记录。

## 4.3 攻击路径推演流

1. 用户选择分析范围（全图/子图/标签过滤）与最大跳数 `maxHops`。
2. 后端从 `ThreatPoint -> AssetNode` 起点集出发进行图遍历。
3. 生成候选路径并去重（路径签名：起点+边序列+终点）。
4. 执行启发式评分，输出分值、优先级与解释。
5. 结果可选择“仅会话展示”或“保存为 AttackPath 工件”写回 Neo4j。

## 5. 前端图建模与交互设计（对标需求管理体验）

## 5.1 信息架构

- **左侧：对象树/过滤器**
  - 资产分组（系统域、环境、标签）
  - ThreatPoint 列表（按严重度/类别）
- **中间：图画布**
  - 拓扑可视化、路径高亮、批注标识
- **右侧：详情与编辑面板**
  - 选中对象属性
  - ThreatPoint 挂载区
  - 路径评分解释区
- **底部：变更与审查面板**
  - Draft 变更清单
  - 保存按钮与冲突提示

## 5.2 核心交互

1. **节点/边编辑**：
   - 快捷键创建节点、拖拽连线创建关系。
   - 属性编辑支持“即时校验 + 延迟保存”。
2. **ThreatPoint 挂载**：
   - 在资产详情直接“挂载威胁点”。
   - 可查看该 ThreatPoint 触发的历史路径。
3. **路径审查模式**：
   - 一键生成路径后，按优先级分组展示。
   - 选中路径时高亮 hop 序列，右侧显示解释（如跨越高价值资产、低信任边界）。
4. **差异可见性**：
   - 未保存变更以颜色/徽标标注。
   - 提交前提供“变更审查摘要”。

## 5.3 可解释评分展示（工程评审导向）

前端应始终把分值拆解为人能理解的因子，而不是只给总分：

- 资产关键性贡献（终点/途径资产的 criticality）
- 跳数惩罚或奖励（过长路径可降低可执行优先级）
- 信任边界穿越次数
- ThreatPoint 基线严重度
- 关系类型权重（如管理面连接高于业务面连接）

展示方式建议：
- 总分 + 优先级标签（P1/P2/P3）
- 因子条形图
- “解释句子”列表（可复制到评审报告）

## 6. API 设计建议（接口分层）

- `POST /imports/excel/preview`：上传并返回校验/映射预览
- `POST /imports/excel/commit`：确认导入并落库
- `GET /graph`：获取当前图与版本号
- `POST /graph/changeset/validate`：校验草稿变更
- `POST /graph/changeset/commit`：提交草稿变更
- `POST /analysis/attack-paths/run`：执行路径推演
- `POST /analysis/attack-paths/save`：将结果保存为 AttackPath
- `GET /analysis/attack-paths`：查询历史分析批次与结果

---

## 7. 落地建议（先小步闭环）

1. **Phase 1：建模与导入闭环**
   - 完成 Asset/Edge/ThreatPoint 导入、编辑、保存。
2. **Phase 2：路径推演闭环**
   - 完成 maxHops 遍历 + 基础评分 + 前端路径高亮。
3. **Phase 3：审查体验增强**
   - 差异审查、解释模板、报告导出、版本对比。

通过以上分层，系统可在不引入重型安全语义框架的前提下，稳定支持“建模—推演—评审”的工程流程。
